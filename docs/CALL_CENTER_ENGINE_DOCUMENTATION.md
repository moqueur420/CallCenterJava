# Подробная документация по `CallCenterEngine.java`

## 1. Назначение класса

Класс `engine.CallCenterEngine` является центральным компонентом всей симуляции. Он управляет:

- внутренним временем модели;
- очередью событий;
- очередью ожидающих заявок;
- активными операторами;
- генерацией входящего потока заявок;
- завершением обслуживания;
- расчётом итоговых метрик.

Если смотреть на архитектуру проекта целиком, то именно `CallCenterEngine` реализует бизнес-логику модели, а остальные классы в основном:

- описывают данные;
- конфигурируют запуск;
- упрощают интеграцию;
- сохраняют результаты.

## 2. Расположение в проекте

Файл находится по пути:

- `src/engine/CallCenterEngine.java`

Связанные классы:

- `config.SimulationConfig`
- `engine.SimEvent`
- `model.CallRequest`
- `model.CallOperator`
- `model.RequestStatus`
- `result.SimulationSnapshot`
- `result.SimulationResults`

## 3. Роль в жизненном цикле приложения

Обычно класс используется не напрямую из UI, а через `AnyLogicAdapter`, но фактическая последовательность работы выглядит так:

```text
1. Создание CallCenterEngine(config)
2. Вызов initialize()
3. Повторный вызов runNextInterval() до тех пор, пока метод возвращает true
4. Вызов getSnapshot() при необходимости пошагового мониторинга
5. Вызов generateResults() после завершения симуляции
```

## 4. Основные идеи модели

`CallCenterEngine` построен на трёх ключевых механизмах:

1. Дискретно-событийная обработка.
   Время двигается не по секундам, а скачками от одного события к другому.

2. Поинтервальная конфигурация.
   Для каждого интервала задаются интенсивность входящего потока и число новых операторов.

3. Сочетание очереди событий и очереди ожидания.
   Очередь событий отвечает за будущие `ARRIVAL` и `DEPARTURE`, а очередь ожидания хранит клиентов, для которых не нашлось свободного оператора.

## 5. Внутреннее состояние класса

В классе есть набор полей, которые полностью описывают состояние текущего прогона.

### 5.1. Конфигурация и генератор случайности

| Поле | Тип | Назначение |
| --- | --- | --- |
| `config` | `SimulationConfig` | входные параметры модели |
| `random` | `Random` | генератор случайных чисел, инициализированный через `config.seed` |

### 5.2. Состояние симуляционного времени

| Поле | Тип | Назначение |
| --- | --- | --- |
| `currentTime` | `double` | текущее время модели в секундах |
| `currentIntervalIndex` | `int` | индекс текущего интервала |

### 5.3. Счётчики идентификаторов

| Поле | Тип | Назначение |
| --- | --- | --- |
| `lastRequestId` | `int` | последний выданный идентификатор заявки |
| `lastOperatorId` | `int` | последний выданный идентификатор оператора |

### 5.4. Основные структуры данных

| Поле | Тип | Назначение |
| --- | --- | --- |
| `eventQueue` | `PriorityQueue<SimEvent>` | будущие события, отсортированные по времени |
| `requestQueue` | `LinkedList<CallRequest>` | очередь ожидающих клиентов |
| `activeOperators` | `List<CallOperator>` | операторы, которые ещё доступны системе |
| `allOperators` | `List<CallOperator>` | все операторы, созданные за прогон |
| `allRequests` | `List<CallRequest>` | все заявки, созданные за прогон |

## 6. Инварианты и неявные предположения

Класс работает корректно только при соблюдении нескольких условий.

### 6.1. Предположения о конфигурации

- `lambdaByInterval` не равен `null`;
- `operatorsSchedule` не равен `null`;
- длины этих списков согласованы с `config.totalIntervals`;
- значения `lambda` положительные;
- `intervalLength > 0`;
- `shiftLength > 0`;
- `mu > 0` и `nu > 0`.

В коде нет явной валидации этих условий, поэтому ответственность лежит на вызывающей стороне.

### 6.2. Предположения об идентификаторах заявок

При завершении обслуживания заявка находится так:

```text
allRequests.get(op.getCurrentRequestId() - 1)
```

Это означает, что модель полагается на инвариант:

- идентификаторы заявок начинаются с `1`;
- идентификаторы растут строго на `1`;
- заявки всегда добавляются в `allRequests` в порядке создания.

Если когда-нибудь логика генерации идентификаторов изменится, эта часть станет хрупкой.

### 6.3. Предположения о потоке выполнения

Класс не является потокобезопасным. Он предполагает одиночный поток исполнения и последовательные вызовы методов.

## 7. Конструктор

## `CallCenterEngine(SimulationConfig config)`

Конструктор:

- сохраняет ссылку на конфигурацию;
- создаёт генератор случайных чисел `Random(config.seed)`.

Что важно:

- объект можно создать, даже если конфигурация неполная;
- фактические ошибки из-за некорректной конфигурации проявятся позже, во время `initialize()` или `runNextInterval()`.

## 8. Метод `initialize()`

Метод подготавливает модель к первому шагу.

### Что делает

1. Устанавливает `currentTime = 0.0`.
2. Устанавливает `currentIntervalIndex = 0`.
3. Добавляет операторов, которые должны начать работу в первом интервале.
4. Генерирует все поступления заявок первого интервала.

### Что не делает

- не очищает списки `allRequests`, `allOperators`, `eventQueue`, `requestQueue`;
- не сбрасывает `lastRequestId` и `lastOperatorId`.

Это значит, что класс в текущем виде не рассчитан на повторное переиспользование одного и того же экземпляра для нескольких независимых прогонов без пересоздания объекта.

## 9. Метод `runNextInterval()`

Это главный шаговый метод всей модели.

### Сигнатура

```java
public boolean runNextInterval()
```

### Возвращаемое значение

- `false`, если все интервалы уже обработаны;
- `true`, если очередной шаг успешно выполнен.

### Полная логика метода

#### Шаг 1. Проверка завершения

Если:

```text
currentIntervalIndex >= config.totalIntervals
```

метод сразу возвращает `false`.

#### Шаг 2. Вычисление конца текущего интервала

Используется формула:

```text
intervalEndTime = (currentIntervalIndex + 1) * config.intervalLength
```

#### Шаг 3. Обработка событий внутри интервала

Пока в `eventQueue` есть события и ближайшее событие происходит не позже конца интервала:

- событие извлекается через `poll()`;
- `currentTime` переводится в момент этого события;
- выполняется ветка по типу события.

Поддерживаются два типа:

- `ARRIVAL` -> `processArrival(event.request)`
- `DEPARTURE` -> `processDeparture(event.operatorId)`

#### Шаг 4. Завершение интервала

После обработки всех событий интервала:

- `currentTime` принудительно сдвигается в точный конец интервала;
- `currentIntervalIndex` увеличивается на `1`.

#### Шаг 5. Подготовка следующего интервала или завершение модели

Если интервалы ещё остались:

- добавляются новые операторы для следующего интервала;
- генерируются заявки следующего интервала.

Если текущий интервал был последним:

- вызывается `finalizeRemainingEvents()`.

## 10. Метод `generateArrivalsForInterval(int intervalIndex)`

Метод отвечает за генерацию всех будущих событий прибытия внутри одного интервала.

### Входные данные

- `intervalIndex` — номер интервала, для которого нужно сгенерировать поток заявок.

### Алгоритм

1. Берётся `lambda = config.lambdaByInterval.get(intervalIndex)`.
2. Вычисляются:
   - начало интервала;
   - конец интервала;
   - локальная переменная `time`, изначально равная началу интервала.
3. В цикле генерируется экспоненциальный интервал между приходами:

```text
time += -ln(1 - u) / lambda
```

4. Если следующее время вышло за границу интервала, цикл завершается.
5. Иначе:
   - создаётся новая заявка;
   - ей присваивается новый `id`;
   - генерируется время терпения;
   - заявка добавляется в `allRequests`;
   - в `eventQueue` добавляется событие `ARRIVAL`.

### Результат

После выполнения метода в очереди событий находятся все прибытия, относящиеся к указанному интервалу.

### Особенности

- заявки создаются заранее, а не в момент реального наступления события;
- `eventQueue` хранит уже готовые объекты `CallRequest`;
- при `lambda = 0` формула становится некорректной.

## 11. Метод `processArrival(CallRequest req)`

Метод вызывается, когда заявка фактически "входит" в систему.

### Алгоритм

1. Выполняется `lazyQueueCleanup()` для удаления клиентов, которые уже должны были уйти.
2. Ищется лучший доступный оператор через `findOptimalOperator()`.
3. Если оператор найден:
   заявка сразу отправляется в обслуживание через `assignRequestToOperator(req, bestOp)`.
4. Если оператора нет:
   заявка ставится в очередь ожидания `requestQueue`.

### Поведенческий смысл

Система делает попытку немедленного обслуживания каждой входящей заявки, не помещая её в очередь без необходимости.

## 12. Метод `processDeparture(int operatorId)`

Метод вызывается при завершении обслуживания у конкретного оператора.

### Алгоритм

1. Оператор ищется в `activeOperators` по `operatorId`.
2. Если оператор не найден, метод завершает работу.
3. Из `allRequests` извлекается текущая заявка оператора.
4. Для заявки вызывается `completeService(currentTime)`.
5. Оператор освобождается через `freeUp()`.
6. Если смена оператора уже завершилась:
   оператор удаляется из `activeOperators`, метод заканчивается.
7. Иначе:
   - выполняется `lazyQueueCleanup()`;
   - если очередь не пуста, оператор сразу получает следующую заявку.

### Почему оператор может не найтись

Такая ситуация может возникнуть, если оператор уже был исключён из `activeOperators` раньше. В этом случае событие просто игнорируется.

### Важная деталь

Этот метод не создаёт новых событий прибытия. Он только:

- закрывает текущее обслуживание;
- потенциально запускает следующее обслуживание для того же оператора.

## 13. Метод `assignRequestToOperator(CallRequest req, CallOperator op)`

Это один из самых важных методов движка. Он связывает клиента и оператора и планирует окончание обслуживания.

### Алгоритм

1. Генерируется длительность обслуживания.
2. Проверяется, может ли оператор закончить обслуживание до конца смены:

```text
op.canTakeRequest(currentTime, serviceTime)
```

3. Если не может:
   - оператор удаляется из `activeOperators`;
   - заявка возвращается в начало очереди;
   - вызывается `processDeparture(op.getId())`;
   - метод завершает работу.
4. Если может:
   - заявка переводится в состояние обслуживания;
   - оператор получает текущую заявку;
   - занятость оператора раскладывается по интервалам;
   - планируется событие `DEPARTURE`.

### Почему заявка возвращается в начало очереди

Если оператор не может взять заявку из-за ограничения по смене, сама заявка не должна терять приоритет. Поэтому она возвращается в начало очереди, а не в конец.

### Тонкость реализации

Ветвь:

```text
activeOperators.remove(op)
requestQueue.addFirst(req)
processDeparture(op.getId())
```

выглядит нетривиально. После удаления из `activeOperators` метод `processDeparture()` уже не найдёт оператора и сразу вернётся. Фактически это означает:

- оператор тихо выводится из системы;
- заявка остаётся в очереди;
- дополнительных действий не происходит.

С точки зрения поведения это работает, но вызов `processDeparture()` в такой форме выглядит лишним и может вводить в заблуждение при чтении кода.

## 14. Метод `lazyQueueCleanup()`

Метод отвечает за "ленивое" удаление ушедших клиентов из очереди.

### Почему cleanup называется ленивым

Потому что отказ клиента не планируется отдельным событием. Вместо этого система время от времени проходит по очереди и проверяет, кто уже должен был уйти.

### Алгоритм

1. Создаётся итератор по `requestQueue`.
2. Для каждой заявки проверяется условие:

```text
currentTime >= appearanceTime + patienceTime
```

3. Если оно выполняется:
   - заявка получает статус `ABANDONED`;
   - временем ухода фиксируется `appearanceTime + patienceTime`;
   - заявка удаляется из очереди.

### Последствия такого подхода

- модель проще, потому что не нужно поддерживать ещё один тип события;
- реальное удаление из очереди может происходить позже, чем математический момент отказа;
- однако в самой заявке время отказа фиксируется корректно.

## 15. Метод `findOptimalOperator()`

Метод выбирает оператора, которому лучше всего назначить новую заявку.

### Что считается оптимумом

Из `activeOperators` выбирается оператор, который:

- не занят;
- имеет минимальное значение `releaseTime`.

### Практический смысл

Для свободных операторов `releaseTime` обычно совпадает с моментом, когда они в последний раз освободились. Фактически движок предпочитает самого давно свободного оператора.

### Если операторов нет

Возвращается `null`, и вызывающий код отправляет заявку в очередь.

## 16. Метод `addOperatorsForInterval(int intervalIndex)`

Метод добавляет новых операторов, стартующих в указанном интервале.

### Алгоритм

1. Берётся число новых операторов из `config.operatorsSchedule`.
2. Для каждого:
   - увеличивается `lastOperatorId`;
   - создаётся `CallOperator(lastOperatorId, currentTime, config.shiftLength)`;
   - оператор добавляется в `activeOperators`;
   - оператор добавляется в `allOperators`.

### Важная интерпретация

`operatorsSchedule[i]` не означает итоговое число операторов в интервале. Это именно число новых операторов, которые начинают смену в начале данного интервала.

## 17. Метод `distributeOperatorBusyTime(CallOperator op, double start, double end)`

Метод раскладывает время одной сессии обслуживания по интервалам, если обслуживание пересекает границы интервалов.

### Зачем он нужен

Если заявка началась в одном интервале, а закончилась в другом, всё занятое время нельзя записать в один интервал. Иначе `K_oper` будет посчитан неверно.

### Алгоритм

1. `currentStart` получает значение `start`.
2. Пока `currentStart < end`:
   - определяется интервал по текущему времени;
   - вычисляется ближайшая граница интервала;
   - в `busyTimeByInterval` добавляется кусок времени;
   - `currentStart` сдвигается к следующему участку.

### Пример

Если обслуживание началось на 1190 секунде, а закончилось на 1210 при длине интервала 1200:

- 10 секунд будут записаны в интервал `0`;
- 10 секунд будут записаны в интервал `1`.

## 18. Метод `finalizeRemainingEvents()`

Этот метод используется после завершения последнего интервала модели.

### Что он делает

1. Пока в `eventQueue` есть события:
   - извлекает событие;
   - переводит `currentTime` в его момент;
   - обрабатывает только `DEPARTURE`.
2. После исчерпания очереди вызывает `lazyQueueCleanup()`.

### Что важно

После последнего интервала новые `ARRIVAL` уже не генерируются, поэтому в хвосте очереди практически ожидаются только завершения обслуживания.

### Возможное следствие

Если в очереди остались заявки, которые ещё не успели потерять терпение к моменту последнего `currentTime`, они после `lazyQueueCleanup()` могут остаться в статусе `WAITING`. Именно поэтому в итогах существует `totalUnresolved`.

## 19. Метод `getSnapshot()`

Метод возвращает лёгкий снимок текущего состояния модели.

### Что входит в снимок

- текущее время;
- номер интервала;
- длина очереди;
- число активных операторов;
- число занятых операторов;
- число обслуженных заявок;
- число потерянных заявок.

### Для чего он нужен

Метод полезен при:

- пошаговой симуляции;
- визуализации;
- отладке;
- внешнем мониторинге хода модели.

## 20. Метод `generateResults()`

Это финальный агрегирующий метод, который превращает внутреннее состояние движка в объект `SimulationResults`.

## 20.1. Подготовка списков

Сначала в результат перекладываются:

- `allRequests`;
- `allOperators`.

Затем формируются три выборки:

- `servedRequests`;
- `abandonedRequests`;
- `resolvedRequests = servedRequests + abandonedRequests`.

## 20.2. Базовые итоговые показатели

Считаются:

- `totalArrived`;
- `totalServed`;
- `totalAbandoned`;
- `totalUnresolved`.

Формула:

```text
totalUnresolved = totalArrived - totalServed - totalAbandoned
```

## 20.3. Средние значения

Среднее ожидание:

```text
averageQueueTime = average(queueTime по resolvedRequests)
```

Среднее обслуживание:

```text
averageServiceTime = average(leavingTime - serviceStartTime по servedRequests)
```

## 20.4. Вероятности

```text
serviceProbability = totalServed / totalArrived
abandonmentProbability = totalAbandoned / totalArrived
```

Если заявок нет, обе вероятности принимают значение `0.0`.

## 20.5. Поинтервальные карты

Создаются шесть структур:

- `kClientByInterval`
- `kOperByInterval`
- `stabilityByInterval`
- `operatorsCountByInterval`
- `serviceCapacityByInterval`
- `targetFunctionByInterval`

Они инициализируются как `LinkedHashMap`, чтобы сохранять предсказуемый порядок интервалов.

Назначение дополнительных карт:

- `stabilityByInterval` хранит значение `K_уст` по интервалам;
- `operatorsCountByInterval` хранит число операторов, покрывающих начало интервала;
- `serviceCapacityByInterval` хранит суммарную интенсивность обслуживания `opsInInterval * mu`.

## 20.6. Группировка заявок по интервалам поступления

Движок собирает `reqsByInterval`, где ключ — это `intervalIndex` заявки, а значение — список завершённых заявок, пришедших в этом интервале.

Обратите внимание: группировка идёт по интервалу поступления заявки, а не по интервалу завершения её обслуживания.

## 20.7. Расчёт `K_client`

Для каждого интервала:

1. Берутся завершённые заявки, поступившие в этом интервале.
2. Считается их среднее ожидание `avgQueue`.
3. Вычисляется:

```text
K_client = avgQueue * config.nu
```

## 20.8. Расчёт `K_oper`

Для каждого интервала:

1. Суммируется занятость всех операторов в этом интервале.
2. Считается число операторов, чья смена покрывает начало интервала.
3. Вычисляется коэффициент загрузки:

```text
K_oper = totalBusyTime / (opsInInterval * config.intervalLength)
```

Если операторов нет, значение равно `0`.

## 20.9. Расчёт коэффициента устойчивости

После подсчёта `opsInInterval` движок получает промежуточную величину:

```text
serviceCapacity = opsInInterval * config.mu
```

Далее вызывается `calculateStability(opsInInterval, lambda)`.

Логика метода:

```text
если lambda <= 0, вернуть INF
если operatorsInInterval == 0, вернуть 0
иначе вернуть operatorsInInterval * mu / lambda
```

То есть итоговая формула для интервала:

```text
K_уст = (opsInInterval * mu) / lambda
```

Дополнительно в `generateResults()`:

- `operatorsCountByInterval.put(i, opsInInterval)` сохраняет число операторов для отчёта;
- `serviceCapacityByInterval.put(i, serviceCapacity)` сохраняет числитель формулы;
- `stabilityByInterval.put(i, stability)` сохраняет сам коэффициент устойчивости;
- `minimumStability` обновляется только для интервалов, где `lambda > 0`;
- `systemStable` становится `false`, если для такого интервала `K_уст < 1`.

Именно эти значения затем используются `SimulationReportWriter` для построения подробного текстового блока с расчётом устойчивости.

## 20.10. Расчёт целевой функции

В коде используются фиксированные коэффициенты:

```text
C1 = 1.0
C2 = 1.0
```

После этого:

```text
targetFunction = C1 * K_oper - C2 * K_client
```

## 21. Взаимодействие внутренних структур

Ниже коротко показано, как связаны основные контейнеры движка.

### `eventQueue`

Хранит будущие события:

- поступление заявок;
- завершение обслуживания.

### `requestQueue`

Хранит только ожидающие заявки, которые:

- уже прибыли;
- но ещё не начали обслуживание;
- и ещё не ушли по таймауту.

### `activeOperators`

Хранит операторов, которых система ещё считает потенциально пригодными для назначения заявок.

### `allOperators`

Нужен не для онлайновой логики, а для итоговой аналитики и отчётности.

### `allRequests`

Тоже нужен и для операционной логики, и для итоговых расчётов.

## 22. Сложность основных операций

Это не формальный анализ, а практическая оценка по используемым структурам.

| Операция | Оценка |
| --- | --- |
| извлечение ближайшего события из `PriorityQueue` | `O(log n)` |
| добавление события в `PriorityQueue` | `O(log n)` |
| добавление заявки в очередь ожидания | `O(1)` |
| выбор оператора через stream/min | `O(m)` |
| cleanup очереди ожидания | `O(q)` |
| поиск оператора по `id` в `activeOperators` | `O(m)` |

Где:

- `n` — число событий;
- `m` — число активных операторов;
- `q` — длина очереди ожидания.

## 23. Сильные стороны текущей реализации

- компактный и читаемый код;
- ясное разделение между доменной моделью и движком;
- отсутствие лишних зависимостей;
- возможность как пошагового, так и полного запуска;
- корректное распределение занятости по интервалам.

## 24. Ограничения и потенциальные слабые места

### 24.1. Нет валидации входных данных

Движок доверяет конфигурации и не защищён от:

- `null`-списков;
- несовпадающих размеров массивов;
- нулевых или отрицательных интенсивностей.

### 24.2. Нет отдельного события отказа клиента

Подход с `lazyQueueCleanup()` упрощает код, но делает модель менее строго событийной.

### 24.3. Повторный запуск одного экземпляра не предусмотрен

`initialize()` не очищает все накопленные коллекции и счётчики.

### 24.4. Поиск по спискам линейный

Для крупных моделей:

- поиск оператора по `id`;
- подбор свободного оператора;
- подсчёты в `generateResults()`

могут стать узким местом.

### 24.5. Логика в `assignRequestToOperator()` требует рефакторинга

Вызов `processDeparture()` после удаления оператора из `activeOperators` семантически неочевиден и усложняет чтение кода.

## 25. Рекомендации по улучшению

- добавить метод валидации `SimulationConfig` перед запуском;
- разделить флаг случайного обслуживания и случайного терпения;
- заменить линейный поиск оператора по `id` на `Map<Integer, CallOperator>`;
- сделать отдельный тип события для отказа клиента;
- позволить безопасно переиспользовать один экземпляр движка;
- вынести веса целевой функции в конфигурацию;
- упростить ветку удаления оператора в `assignRequestToOperator()`.

## 26. Краткий вывод

`CallCenterEngine` — это полноценный симуляционный движок, который:

- создаёт поток заявок;
- управляет очередью ожидания;
- распределяет нагрузку по операторам;
- завершает обслуживание;
- агрегирует статистику.

Несмотря на компактность, класс содержит почти всю содержательную логику проекта. Если нужно понять, как работает модель колл-центра, начинать чтение кода следует именно с него.
