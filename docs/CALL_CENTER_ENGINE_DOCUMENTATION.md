# Подробная документация по `CallCenterEngine.java`

## 1. Назначение класса

Класс `engine.CallCenterEngine` является центральным компонентом всей симуляции. Он управляет:

- внутренним временем модели;
- очередью событий;
- очередью ожидающих заявок;
- активными операторами;
- генерацией входящего потока заявок;
- подготовкой и нормализацией конфигурации перед запуском;
- автоматическим построением расписания операторов, если оно не задано явно;
- завершением обслуживания;
- расчётом итоговых метрик.

Если смотреть на архитектуру проекта целиком, то именно `CallCenterEngine` реализует бизнес-логику модели, а остальные классы в основном:

- описывают данные;
- конфигурируют запуск;
- упрощают интеграцию;
- сохраняют результаты.

## 3. Роль в жизненном цикле приложения

Обычно класс используется не напрямую из UI, а через `AnyLogicAdapter`, но фактическая последовательность работы выглядит так:

```text
1. Подготовка или загрузка SimulationConfig
2. Создание CallCenterEngine(config)
3. Вызов initialize()
4. Нормализация конфигурации внутри движка
5. Повторный вызов runNextInterval() до тех пор, пока метод возвращает true
6. Вызов getSnapshot() при необходимости пошагового мониторинга
7. Вызов generateResults() после завершения симуляции
```

Если конфигурация приходит из `DatasetConfigLoader`, движок обычно получает:

- `totalOperators`;
- `totalIntervals`;
- `intervalLength`;
- `arrivalsByInterval`.

После этого он сам может достроить:

- `lambdaByInterval`;
- `operatorsSchedule`.

## 4. Основные идеи модели

`CallCenterEngine` построен на четырёх ключевых механизмах:

1. Дискретно-событийная обработка.
   Время двигается не по секундам, а скачками от одного события к другому.

2. Поинтервальная конфигурация.
   Нагрузка и запуск смен трактуются по интервалам фиксированной длины.

3. Предварительная подготовка конфигурации.
   Перед первым шагом движок валидирует входные данные и при необходимости строит недостающие `lambdaByInterval` и `operatorsSchedule`.

4. Сочетание очереди событий и очереди ожидания.
   Очередь событий отвечает за будущие `ARRIVAL` и `DEPARTURE`, а очередь ожидания хранит клиентов, для которых не нашлось свободного оператора.

## 5. Внутреннее состояние класса

В классе есть набор полей, которые полностью описывают состояние текущего прогона.

### 5.1. Конфигурация и генератор случайности

| Поле | Тип | Назначение |
| --- | --- | --- |
| `config` | `SimulationConfig` | входные параметры модели; часть полей может быть дополнена самим движком |
| `random` | `Random` | генератор случайных чисел, инициализированный через `config.seed` |

### 5.2. Состояние симуляционного времени

| Поле | Тип | Назначение |
| --- | --- | --- |
| `currentTime` | `double` | текущее время модели в секундах |
| `currentIntervalIndex` | `int` | индекс текущего интервала |

### 5.3. Счётчики идентификаторов

| Поле | Тип | Назначение |
| --- | --- | --- |
| `lastRequestId` | `int` | последний выданный идентификатор заявки |
| `lastOperatorId` | `int` | последний выданный идентификатор оператора |

### 5.4. Основные структуры данных

| Поле | Тип | Назначение |
| --- | --- | --- |
| `eventQueue` | `PriorityQueue<SimEvent>` | будущие события, отсортированные по времени |
| `requestQueue` | `LinkedList<CallRequest>` | очередь ожидающих клиентов |
| `activeOperators` | `List<CallOperator>` | операторы, которые ещё доступны системе |
| `allOperators` | `List<CallOperator>` | все операторы, созданные за прогон |
| `allRequests` | `List<CallRequest>` | все заявки, созданные за прогон |

## 6. Подготовка конфигурации и инварианты

Текущая версия движка больше не рассчитывает на полностью подготовленную конфигурацию со стороны вызывающего кода. Перед запуском она сама валидирует и, при необходимости, достраивает часть параметров.

### 6.1. Общая схема подготовки

Во время `initialize()` вызывается цепочка:

```text
prepareSimulationConfig()
  -> validateCoreConfig()
  -> ensureArrivalsByInterval()
  -> ensureLambdaByInterval()
  -> ensureOperatorsSchedule()
```

### 6.2. `validateCoreConfig()`

Движок проверяет базовые ограничения:

- `totalOperators >= 0`;
- `totalIntervals >= 0`;
- `intervalLength > 0`;
- `shiftLength > 0`;
- `mu > 0`;
- `nu > 0`.

При нарушении любого условия выбрасывается `IllegalArgumentException`.

### 6.3. `ensureArrivalsByInterval()`

Если задан `arrivalsByInterval`, движок проверяет:

- длина списка совпадает с `config.totalIntervals`;
- элементы не равны `null`;
- значения неотрицательные.

### 6.4. `ensureLambdaByInterval()`

Поддерживаются два сценария.

1. `lambdaByInterval` уже задан.
   Тогда движок валидирует размер списка и проверяет, что все значения неотрицательные.

2. `lambdaByInterval == null`, но задан `arrivalsByInterval`.
   Тогда движок автоматически строит:

```text
lambda[i] = arrivalsByInterval[i] / intervalLength
```

Если не задан ни `lambdaByInterval`, ни `arrivalsByInterval`, симуляция не стартует.

### 6.5. `ensureOperatorsSchedule()`

Если `operatorsSchedule` задан явно, движок проверяет:

- длину списка;
- отсутствие `null`;
- неотрицательность значений;
- ограничение:

```text
sum(operatorsSchedule) <= totalOperators
```

Если `operatorsSchedule == null`, движок строит расписание автоматически.

### 6.6. Автоматическое построение расписания операторов

Это новый крупный функциональный блок.

Логика состоит из двух фаз:

1. `buildMinimalCoverageSchedule(...)`
   Движок проходит по интервалам слева направо и добавляет столько новых смен, сколько нужно для покрытия:

```text
requiredOperators[i] = ceil(lambda[i] / mu)
```

при учёте того, что одна смена покрывает:

```text
shiftCoverageIntervals = ceil(shiftLength / intervalLength)
```

2. `buildBestEffortSchedule(...)`
   Если минимальное покрытие требует больше операторов, чем разрешает `totalOperators`, движок распределяет доступных операторов эвристически:

- выбирает лучший старт смены по максимальному покрытию недостающего спроса;
- при равенстве предпочитает вариант с большим остаточным давлением;
- при полном равенстве берёт более поздний интервал;
- затем локально пытается улучшить расписание переносами через `improveScheduleLocally()`.

Качество расписания оценивается функцией:

```text
coverageScore = sum(min(activeOperators[i], requiredOperators[i]))
```

Итоговое авто-расписание:

- всегда имеет длину `totalIntervals`;
- не содержит отрицательных значений;
- не использует больше операторов, чем разрешено `totalOperators`.

### 6.7. Инварианты, которые всё ещё важны

При завершении обслуживания заявка ищется так:

```text
allRequests.get(op.getCurrentRequestId() - 1)
```

Это означает, что движок по-прежнему зависит от инварианта:

- идентификаторы заявок начинаются с `1`;
- растут без пропусков;
- `allRequests` хранит заявки в порядке создания.

Кроме того, класс не является потокобезопасным и рассчитан на последовательные вызовы из одного потока.

## 7. Конструктор

## `CallCenterEngine(SimulationConfig config)`

Конструктор:

- сохраняет ссылку на конфигурацию;
- создаёт генератор случайных чисел `Random(config.seed)`.

Что важно:

- объект можно создать и с неполной конфигурацией;
- фактическая валидация и нормализация происходят в `initialize()`.

## 8. Метод `initialize()`

Метод подготавливает модель к первому шагу.

### Что делает

1. Вызывает `prepareSimulationConfig()`.
2. Устанавливает `currentTime = 0.0`.
3. Устанавливает `currentIntervalIndex = 0`.
4. Добавляет операторов, которые должны начать работу в первом интервале.
5. Генерирует все поступления заявок первого интервала.

### Что не делает

- не очищает списки `allRequests`, `allOperators`, `eventQueue`, `requestQueue`;
- не сбрасывает `lastRequestId` и `lastOperatorId`.

Это значит, что экземпляр движка по-прежнему не рассчитан на повторное переиспользование для нескольких независимых прогонов.

## 9. Метод `runNextInterval()`

Это главный шаговый метод всей модели.

### Сигнатура

```java
public boolean runNextInterval()
```

### Возвращаемое значение

- `false`, если все интервалы уже обработаны;
- `true`, если очередной шаг успешно выполнен.

### Полная логика метода

#### Шаг 1. Проверка завершения

Если:

```text
currentIntervalIndex >= config.totalIntervals
```

метод сразу возвращает `false`.

#### Шаг 2. Вычисление конца текущего интервала

Используется формула:

```text
intervalEndTime = (currentIntervalIndex + 1) * config.intervalLength
```

#### Шаг 3. Обработка событий внутри интервала

Пока в `eventQueue` есть события и ближайшее событие происходит не позже конца интервала:

- событие извлекается через `poll()`;
- `currentTime` переводится в момент этого события;
- выполняется ветка по типу события.

Поддерживаются два типа:

- `ARRIVAL` -> `processArrival(event.request)`
- `DEPARTURE` -> `processDeparture(event.operatorId)`

#### Шаг 4. Завершение интервала

После обработки всех событий интервала:

- `currentTime` принудительно сдвигается в точный конец интервала;
- `currentIntervalIndex` увеличивается на `1`.

#### Шаг 5. Подготовка следующего интервала или завершение модели

Если интервалы ещё остались:

- добавляются новые операторы для следующего интервала;
- генерируются заявки следующего интервала.

Если текущий интервал был последним:

- вызывается `finalizeRemainingEvents()`.

## 10. Метод `generateArrivalsForInterval(int intervalIndex)`

Метод отвечает за генерацию всех будущих событий прибытия внутри одного интервала.

### Поддерживаемые режимы

1. Если задан `arrivalsByInterval`, используется точное количество заявок на интервал.
2. Иначе используется `lambdaByInterval` и пуассоновская генерация межприходов.

### Поведение при `arrivalsByInterval`

Если `config.arrivalsByInterval != null`, движок вызывает:

```text
generateFixedArrivalsForInterval(intervalIndex, arrivalsByInterval[intervalIndex])
```

В этом режиме число заявок фиксировано, а моменты появления равномерно распределяются внутри интервала и затем сортируются.

### Поведение при `lambdaByInterval`

Если фиксированного числа заявок нет, берётся:

```text
lambda = config.lambdaByInterval.get(intervalIndex)
```

Если `lambda <= 0`, новых заявок в интервале не создаётся.

Иначе используется формула:

```text
time += -ln(1 - u) / lambda
```

и для каждого подходящего момента вызывается `enqueueRequest(...)`.

### Важная тонкость

Если одновременно заданы и `arrivalsByInterval`, и `lambdaByInterval`, фактическая генерация заявок идёт по `arrivalsByInterval`, а `lambdaByInterval` остаётся источником для аналитики и автопостроения расписания.

## 11. Вспомогательные методы генерации заявок

### `generateFixedArrivalsForInterval(int intervalIndex, int arrivalsCount)`

Метод:

1. создаёт ровно `arrivalsCount` случайных моментов прихода внутри интервала;
2. сортирует их;
3. для каждого момента вызывает `enqueueRequest(...)`.

Это даёт детерминированное число заявок на интервал при случайном распределении внутри интервала.

### `enqueueRequest(double arrivalTime, int intervalIndex)`

Метод:

1. увеличивает `lastRequestId`;
2. вычисляет терпение клиента;
3. создаёт `CallRequest`;
4. добавляет заявку в `allRequests`;
5. планирует событие `ARRIVAL`.

Терпение клиента вычисляется так:

```text
если useRandomServiceTime == true:
    patience = -ln(1 - u) / nu
иначе:
    patience = 1 / nu
```

Флаг `useRandomServiceTime` влияет не только на длительность обслуживания, но и на терпение клиента.

## 12. Метод `processArrival(CallRequest req)`

Метод вызывается, когда заявка фактически входит в систему.

### Алгоритм

1. Выполняется `lazyQueueCleanup()`.
2. Выполняется попытка немедленного назначения через `tryAssignRequest(req)`.
3. Если подходящего оператора нет, заявка ставится в хвост `requestQueue`.

### Поведенческий смысл

Система сначала пытается обслужить заявку сразу, и только при неудаче отправляет её в очередь.

## 13. Метод `processDeparture(int operatorId)`

Метод вызывается при завершении обслуживания у конкретного оператора.

### Алгоритм

1. Оператор ищется в `activeOperators` по `operatorId`.
2. Если оператор не найден, метод завершает работу.
3. Из `allRequests` извлекается текущая заявка оператора.
4. Для заявки вызывается `completeService(currentTime)`.
5. Оператор освобождается через `freeUp()`.
6. Вызывается `removeExpiredIdleOperators()`.
7. Запускается `assignQueuedRequests()`.

### Почему оператор может не найтись

Такая ситуация возможна, если оператор уже был выведен из `activeOperators` как простаивающий после конца смены. В этом случае событие безопасно игнорируется.

## 14. Методы назначения заявки оператору

### `tryAssignRequest(CallRequest req)`

Метод:

1. сэмплирует длительность обслуживания через `sampleServiceTime()`;
2. ищет подходящего оператора через `findOptimalOperator(serviceTime)`;
3. если оператор найден, назначает заявку и возвращает `true`;
4. иначе возвращает `false`.

Важный нюанс: длительность обслуживания сэмплируется в момент попытки назначения, а не закрепляется за заявкой заранее. Поэтому одна и та же заявка в очереди может получать разные случайные `serviceTime` при разных попытках назначения, пока не будет реально принята оператором.

### `assignRequestToOperator(CallRequest req, CallOperator op, double serviceTime)`

Текущая версия метода больше не содержит ветки с возвратом заявки в начало очереди и искусственным вызовом `processDeparture()`.

Теперь метод делает только прямое назначение:

1. переводит заявку в `IN_SERVICE`;
2. записывает `operatorId` и время начала обслуживания;
3. переводит оператора в занятое состояние;
4. распределяет занятость по интервалам;
5. планирует `DEPARTURE` на момент `currentTime + serviceTime`.

### `assignQueuedRequests()`

Метод разбирает накопившуюся очередь после освобождения оператора или после выхода новой смены.

Алгоритм:

1. выполнить `lazyQueueCleanup()`;
2. если очередь пуста, завершиться;
3. посмотреть заявку в голове очереди;
4. попробовать назначить именно её;
5. если назначение удалось, удалить голову и повторить цикл;
6. если назначение не удалось, остановиться.

Следствие: очередь остаётся строгой FIFO. Движок не перескакивает через голову к более поздним заявкам.

## 15. Методы выбора доступного оператора

### `sampleServiceTime()`

Формула:

```text
если useRandomServiceTime == true:
    serviceTime = -ln(1 - u) / mu
иначе:
    serviceTime = 1 / mu
```

### `findOptimalOperator(double serviceTime)`

Перед поиском вызывается `removeExpiredIdleOperators()`.

Далее из `activeOperators` выбирается оператор, который:

- не занят;
- успевает завершить разговор до конца смены:

```text
currentTime + serviceTime <= shiftEndTime
```

Из подходящих кандидатов выбирается минимум по:

1. `releaseTime`;
2. `shiftEndTime`;
3. `id`.

Практически это означает: движок предпочитает оператора, который давно свободен, а при равенстве старается раньше задействовать того, чья смена закончится раньше.

### `removeExpiredIdleOperators()`

Из `activeOperators` удаляются только те операторы, которые:

- уже не заняты;
- достигли конца смены.

Занятый оператор остаётся в системе до завершения разговора.

## 16. Метод `addOperatorsForInterval(int intervalIndex)`

Метод добавляет новых операторов, стартующих в указанном интервале.

### Алгоритм

1. Берётся число новых операторов из `config.operatorsSchedule`.
2. Для каждого:
   - увеличивается `lastOperatorId`;
   - создаётся `CallOperator(lastOperatorId, currentTime, config.shiftLength)`;
   - оператор добавляется в `activeOperators`;
   - оператор добавляется в `allOperators`.
3. После добавления новых смен вызывается `assignQueuedRequests()`.

### Важная интерпретация

`operatorsSchedule[i]` не означает итоговое число операторов в интервале. Это именно число новых операторов, которые начинают смену в начале данного интервала.

## 17. Метод `distributeOperatorBusyTime(CallOperator op, double start, double end)`

Метод раскладывает время одной сессии обслуживания по интервалам, если обслуживание пересекает границы интервалов.

### Зачем он нужен

Если заявка началась в одном интервале, а закончилась в другом, всё занятое время нельзя записать в один интервал. Иначе `K_oper` будет посчитан неверно.

### Алгоритм

1. `currentStart` получает значение `start`.
2. Пока `currentStart < end`:
   - определяется интервал по текущему времени;
   - вычисляется ближайшая граница интервала;
   - в `busyTimeByInterval` добавляется кусок времени;
   - `currentStart` сдвигается к следующему участку.

### Пример

Если обслуживание началось на 1190 секунде, а закончилось на 1210 при длине интервала 1200:

- 10 секунд будут записаны в интервал `0`;
- 10 секунд будут записаны в интервал `1`.

## 18. Метод `finalizeRemainingEvents()`

Этот метод используется после завершения последнего интервала модели.

### Что он делает

1. Пока в `eventQueue` есть события:
   - извлекает событие;
   - переводит `currentTime` в его момент;
   - обрабатывает только `DEPARTURE`.
2. После исчерпания очереди вызывает `lazyQueueCleanup()`.

### Что важно

После последнего интервала новые `ARRIVAL` уже не генерируются, поэтому в хвосте очереди практически ожидаются только завершения обслуживания.

### Возможное следствие

Если в очереди остались заявки, которые ещё не успели потерять терпение к моменту последнего `currentTime`, они после `lazyQueueCleanup()` могут остаться в статусе `WAITING`. Именно поэтому в итогах существует `totalUnresolved`.

## 19. Метод `getSnapshot()`

Метод возвращает лёгкий снимок текущего состояния модели.

### Что входит в снимок

- текущее время;
- номер интервала;
- длина очереди;
- число активных операторов;
- число занятых операторов;
- число обслуженных заявок;
- число потерянных заявок.

### Для чего он нужен

Метод полезен при:

- пошаговой симуляции;
- визуализации;
- отладке;
- внешнем мониторинге хода модели.

## 20. Метод `generateResults()`

Это финальный агрегирующий метод, который превращает внутреннее состояние движка в объект `SimulationResults`.

## 20.1. Подготовка списков

Сначала в результат перекладываются:

- `allRequests`;
- `allOperators`.

Затем формируются три выборки:

- `servedRequests`;
- `abandonedRequests`;
- `resolvedRequests = servedRequests + abandonedRequests`.

## 20.2. Базовые итоговые показатели

Считаются:

- `totalArrived`;
- `totalServed`;
- `totalAbandoned`;
- `totalUnresolved`.

Формула:

```text
totalUnresolved = totalArrived - totalServed - totalAbandoned
```

## 20.3. Средние значения

Среднее ожидание:

```text
averageQueueTime = average(queueTime по resolvedRequests)
```

Среднее обслуживание:

```text
averageServiceTime = average(leavingTime - serviceStartTime по servedRequests)
```

## 20.4. Вероятности

```text
serviceProbability = totalServed / totalArrived
abandonmentProbability = totalAbandoned / totalArrived
```

Если заявок нет, обе вероятности принимают значение `0.0`.

## 20.5. Поинтервальные карты

Создаются шесть структур:

- `kClientByInterval`
- `kOperByInterval`
- `stabilityByInterval`
- `operatorsCountByInterval`
- `serviceCapacityByInterval`
- `targetFunctionByInterval`

Они инициализируются как `LinkedHashMap`, чтобы сохранять предсказуемый порядок интервалов.

Назначение дополнительных карт:

- `stabilityByInterval` хранит значение `K_уст` по интервалам;
- `operatorsCountByInterval` хранит число операторов, покрывающих начало интервала;
- `serviceCapacityByInterval` хранит суммарную интенсивность обслуживания `opsInInterval * mu`.

## 20.6. Группировка заявок по интервалам поступления

Движок собирает `reqsByInterval`, где ключ — это `intervalIndex` заявки, а значение — список завершённых заявок, пришедших в этом интервале.

Обратите внимание: группировка идёт по интервалу поступления заявки, а не по интервалу завершения её обслуживания.

## 20.7. Расчёт `K_client`

Для каждого интервала:

1. Берутся завершённые заявки, поступившие в этом интервале.
2. Считается их среднее ожидание `avgQueue`.
3. Вычисляется:

```text
K_client = avgQueue * config.nu
```

## 20.8. Расчёт `K_oper`

Для каждого интервала:

1. Суммируется занятость всех операторов в этом интервале.
2. Считается число операторов, чья смена покрывает начало интервала.
3. Вычисляется коэффициент загрузки:

```text
K_oper = totalBusyTime / (opsInInterval * config.intervalLength)
```

Если операторов нет, значение равно `0`.

## 20.9. Расчёт коэффициента устойчивости

После подсчёта `opsInInterval` движок получает промежуточную величину:

```text
serviceCapacity = opsInInterval * config.mu
```

Далее вызывается `calculateStability(opsInInterval, lambda)`.

Логика метода:

```text
если lambda <= 0, вернуть INF
если operatorsInInterval == 0, вернуть 0
иначе вернуть operatorsInInterval * mu / lambda
```

То есть итоговая формула для интервала:

```text
K_уст = (opsInInterval * mu) / lambda
```

Дополнительно в `generateResults()`:

- `operatorsCountByInterval.put(i, opsInInterval)` сохраняет число операторов для отчёта;
- `serviceCapacityByInterval.put(i, serviceCapacity)` сохраняет числитель формулы;
- `stabilityByInterval.put(i, stability)` сохраняет сам коэффициент устойчивости;
- `minimumStability` обновляется только для интервалов, где `lambda > 0`;
- `systemStable` становится `false`, если для такого интервала `K_уст < 1`.

Именно эти значения затем используются `SimulationReportWriter` для построения подробного текстового блока с расчётом устойчивости.

## 20.10. Расчёт целевой функции

В коде используются фиксированные коэффициенты:

```text
C1 = 1.0
C2 = 1.0
```

После этого:

```text
targetFunction = C1 * K_oper - C2 * K_client
```

## 21. Взаимодействие внутренних структур

Ниже коротко показано, как связаны основные контейнеры движка.

### `eventQueue`

Хранит будущие события:

- поступление заявок;
- завершение обслуживания.

### `requestQueue`

Хранит только ожидающие заявки, которые:

- уже прибыли;
- но ещё не начали обслуживание;
- и ещё не ушли по таймауту.

### `activeOperators`

Хранит операторов, которых система ещё считает потенциально пригодными для назначения заявок.

### `allOperators`

Нужен не для онлайновой логики, а для итоговой аналитики и отчётности.

### `allRequests`

Тоже нужен и для операционной логики, и для итоговых расчётов.

## 22. Сложность основных операций

Это не строгий формальный анализ, а практическая оценка по используемым структурам.

| Операция | Оценка |
| --- | --- |
| извлечение ближайшего события из `PriorityQueue` | `O(log n)` |
| добавление события в `PriorityQueue` | `O(log n)` |
| добавление заявки в очередь ожидания | `O(1)` |
| очистка очереди `lazyQueueCleanup()` | `O(q)` |
| выбор подходящего оператора | `O(m)` |
| поиск оператора по `id` в `activeOperators` | `O(m)` |
| построение минимального авто-расписания | `O(T)` |
| greedy-часть best-effort расписания | примерно `O(totalOperators * T * S)` |
| одна итерация локального улучшения расписания | примерно `O(T^2 * S)` |

Где:

- `n` — число событий;
- `m` — число активных операторов;
- `q` — длина очереди ожидания;
- `T` — число интервалов;
- `S` — сколько интервалов покрывает одна смена.

## 23. Сильные стороны текущей реализации

- есть явная валидация и нормализация конфигурации до старта симуляции;
- поддерживаются два режима входного потока: `lambdaByInterval` и `arrivalsByInterval`;
- недостающие `lambdaByInterval` и `operatorsSchedule` могут быть автоматически достроены самим движком;
- авто-расписание уважает ограничение `totalOperators`;
- очередь остаётся строгой FIFO даже при нехватке операторов;
- выбор оператора учитывает не только факт простоя, но и возможность завершить разговор до конца смены;
- поддерживаются и пошаговый режим, и быстрый полный прогон;
- занятость операторов корректно раскладывается по интервалам для расчёта метрик.

## 24. Ограничения и потенциальные слабые места

### 24.1. Повторный запуск одного экземпляра не предусмотрен

`initialize()` не очищает накопленные коллекции и счётчики.

### 24.2. Нет отдельного события отказа клиента

Подход с `lazyQueueCleanup()` упрощает код, но делает модель менее строго событийной.

### 24.3. `totalIntervals == 0` формально проходит базовую валидацию, но практически не поддержан

После валидации движок всё равно обращается к нулевому интервалу через:

- `addOperatorsForInterval(0)`;
- `generateArrivalsForInterval(0)`.

На практике конфигурацию нужно считать корректной только при `totalIntervals > 0`.

### 24.4. При одновременной передаче `arrivalsByInterval` и `lambdaByInterval` возможна рассинхронизация

Фактическая генерация заявок идёт по `arrivalsByInterval`, а аналитика и авто-расписание ориентируются на `lambdaByInterval`.

Если эти источники не согласованы между собой, симуляция и отчёт могут описывать разные сценарии нагрузки.

### 24.5. Сервисное время сэмплируется на каждой попытке назначения

Пока заявка стоит в очереди, при каждой новой попытке назначения ей заново генерируется `serviceTime`. Это влияет на решение "успеет ли оператор до конца смены" и делает поведение модели менее интуитивным, чем при заранее фиксированной длительности обслуживания на заявку.

### 24.6. Поиск по спискам линейный

Для крупных моделей поиск оператора по `id`, подбор лучшего оператора и часть итоговых подсчётов могут стать узким местом.

### 24.7. Веса целевой функции зашиты в код

Коэффициенты `C1` и `C2` пока фиксированы и не приходят из конфигурации.

### 24.8. `minimumStability` может остаться равным `Infinity`

Если во всех интервалах `lambda <= 0`, минимальный коэффициент устойчивости так и остаётся `Double.POSITIVE_INFINITY`.

## 25. Рекомендации по улучшению

- разделить флаг случайного обслуживания и случайного терпения;
- заменить линейный поиск оператора по `id` на `Map<Integer, CallOperator>`;
- сделать отдельный тип события для отказа клиента;
- позволить безопасно переиспользовать один экземпляр движка;
- явно валидировать `totalIntervals > 0`;
- синхронизировать или жёстко запрещать противоречивую передачу `arrivalsByInterval` и `lambdaByInterval`;
- вынести веса целевой функции в конфигурацию;
- при необходимости закреплять `serviceTime` за заявкой в момент её появления, а не при каждой попытке назначения.

## 26. Краткий вывод

`CallCenterEngine` остаётся центральным симуляционным компонентом проекта, но в текущей версии это уже не просто цикл обработки событий. Движок валидирует и достраивает конфигурацию, умеет работать с двумя режимами входного потока, автоматически планирует запуск смен при нехватке явного расписания и только после этого запускает собственно симуляцию.

Если нужно понять актуальное поведение системы, читать код и документацию стоит именно отсюда: здесь сосредоточены правила генерации нагрузки, назначения операторов, ухода клиентов, планирования смен и расчёта итоговых метрик.
