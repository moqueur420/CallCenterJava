# Документация проекта CallCenterJava

## 1. Назначение проекта

`CallCenterJava` моделирует работу колл-центра методом дискретно-событийной симуляции. Проект предназначен для исследования баланса между:

- качеством обслуживания клиентов;
- загрузкой операторов;
- расписанием выхода операторов на смену;
- потерями из-за ухода клиентов из очереди.

Модель работает по временным интервалам фиксированной длины. Для каждого интервала задаются:

- интенсивность входящего потока заявок `lambda`;
- количество новых операторов, начинающих смену.

Внутри интервала события обрабатываются не по тикам, а по реальному времени наступления события. Это делает модель точнее и дешевле по вычислениям по сравнению с пошаговой симуляцией по секундам.

## 2. Предметная область и основные допущения

### 2.1. Что считается заявкой

Заявка (`CallRequest`) представляет одно входящее обращение клиента. Для неё фиксируются:

- момент появления в системе;
- максимально допустимое время ожидания;
- интервал, в котором заявка появилась;
- фактическое время ожидания;
- оператор, который начал обслуживание;
- момент ухода из системы;
- итоговый статус.

### 2.2. Что считается оператором

Оператор (`CallOperator`) начинает смену в определённый момент времени и работает фиксированную продолжительность `shiftLength`. Оператор:

- может быть свободен или занят;
- может обслуживать только одну заявку одновременно;
- после завершения обслуживания либо берёт следующую заявку из очереди, либо покидает активный состав, если смена закончилась.

### 2.3. Как трактуется очередь

Очередь реализована как `LinkedList<CallRequest>` и обслуживается по принципу FIFO:

- новые необслуженные заявки добавляются в хвост очереди;
- свободный оператор берёт заявку из головы очереди;
- если оператор не может принять заявку из-за окончания смены, заявка возвращается в начало очереди.

### 2.4. Что считается уходом клиента

Если клиент ждал дольше своего допустимого времени ожидания (`patienceTime`), заявка получает статус `ABANDONED`. В модели нет отдельного события "клиент ушёл"; этот переход проверяется лениво, когда движок:

- обрабатывает новое поступление заявки;
- завершает обслуживание;
- завершает всю симуляцию.

Это важная особенность поведения модели.

## 4. Жизненный цикл приложения

Ниже показан основной сценарий работы проекта.

```text
demo.Main
  -> создает SimulationConfig
  -> генерирует lambdaByInterval
  -> задает operatorsSchedule
  -> AnyLogicAdapter.runFastFullSimulation(config)
       -> initSimulation(config)
            -> new CallCenterEngine(config)
            -> engine.initialize()
       -> пока engine.runNextInterval() == true
       -> engine.generateResults()
  -> SimulationReportWriter.writeReport(config, results, path)
  -> печать итоговой сводки в консоль
```

## 5. Конфигурация симуляции

Класс `config.SimulationConfig` хранит все входные параметры модели.

| Поле | Значение по умолчанию | Единицы | Смысл |
| --- | --- | --- | --- |
| `mu` | `1.0 / 240.0` | 1/сек | интенсивность обслуживания |
| `nu` | `1.0 / 240.0` | 1/сек | интенсивность потери терпения клиента |
| `intervalLength` | `1200.0` | сек | длительность одного интервала |
| `totalIntervals` | `36` | шт | количество интервалов в модели |
| `shiftLength` | `14400.0` | сек | длительность смены оператора |
| `lambdaByInterval` | `null` | 1/сек | интенсивность входящего потока по интервалам |
| `operatorsSchedule` | `null` | шт | число новых операторов, стартующих в интервале |
| `seed` | `System.currentTimeMillis()` | число | зерно генератора случайных чисел |
| `useRandomServiceTime` | `true` | флаг | включение случайных длительностей |

### 5.1. Интерпретация параметров

- `mu` используется для генерации длительности обслуживания.
- `nu` используется для генерации времени терпения клиента.
- `intervalLength * totalIntervals` задаёт горизонт моделирования.
- `shiftLength` задаёт длину смены каждого нового оператора.
- `lambdaByInterval.size()` и `operatorsSchedule.size()` должны быть согласованы с `totalIntervals`.

### 5.2. Особенность флага `useRandomServiceTime`

Название поля звучит так, как будто оно влияет только на обслуживание, но фактически оно управляет сразу двумя вещами:

- случайной длительностью обслуживания;
- случайным временем терпения клиента.

Если флаг равен `false`, обе величины становятся детерминированными и равны:

- `1 / mu` для обслуживания;
- `1 / nu` для терпения.

## 6. Математическая модель

### 6.1. Поступление заявок

Для каждого интервала заявки генерируются как пуассоновский поток с интенсивностью `lambda`. Время между соседними поступлениями вычисляется по формуле:

```text
delta = -ln(1 - u) / lambda
```

где `u` — равномерная случайная величина на отрезке `(0, 1)`.

### 6.2. Время терпения клиента

Если `useRandomServiceTime == true`, терпение клиента также генерируется экспоненциально:

```text
patience = -ln(1 - u) / nu
```

Иначе:

```text
patience = 1 / nu
```

### 6.3. Время обслуживания

Если `useRandomServiceTime == true`, длительность обслуживания:

```text
serviceTime = -ln(1 - u) / mu
```

Иначе:

```text
serviceTime = 1 / mu
```

## 7. Подробное описание пакетов и классов

## 7.1. Пакет `config`

### `SimulationConfig`

Ответственность класса:

- хранить все параметры одного прогона;
- обеспечивать воспроизводимость за счёт `seed`;
- отделять конфигурацию от логики движка.

Класс является простым контейнером данных и не содержит вычислительной логики.

## 7.2. Пакет `model`

### `CallRequest`

Состояние объекта:

- `id` — уникальный идентификатор заявки;
- `appearanceTime` — время появления заявки;
- `patienceTime` — максимум ожидания перед уходом;
- `intervalIndex` — интервал поступления;
- `queueTime` — фактическое время ожидания;
- `status` — текущий статус из `RequestStatus`;
- `operatorId` — оператор, начавший обслуживание;
- `serviceStartTime` — момент начала обслуживания;
- `leavingTime` — момент выхода из системы.

Основные методы:

| Метод | Назначение |
| --- | --- |
| `startService(double currentTime, int operatorId)` | переводит заявку в `IN_SERVICE`, фиксирует время ожидания и оператора |
| `completeService(double currentTime)` | переводит заявку в `SERVED` и записывает время выхода |
| `abandon(double currentTime)` | переводит заявку в `ABANDONED`, фиксирует уход и время ожидания |

Правила переходов:

```text
WAITING -> IN_SERVICE -> SERVED
WAITING -> ABANDONED
```

### `CallOperator`

Состояние объекта:

- `id` — идентификатор оператора;
- `shiftStartTime` — начало смены;
- `shiftEndTime` — конец смены;
- `busy` — признак занятости;
- `currentRequestId` — текущая заявка;
- `releaseTime` — момент освобождения;
- `busyTimeByInterval` — карта занятости по интервалам.

Основные методы:

| Метод | Назначение |
| --- | --- |
| `canTakeRequest(double currentTime, double estimatedServiceTime)` | проверяет, успеет ли оператор закончить заявку до конца смены |
| `assignRequest(int requestId, double currentTime, double serviceTime)` | помечает оператора занятым и записывает время освобождения |
| `freeUp()` | освобождает оператора после завершения обслуживания |
| `addBusyTime(int interval, double time)` | добавляет накопленное занятое время в указанный интервал |

### `RequestStatus`

Перечисление описывает четыре состояния заявки:

- `WAITING` — заявка находится в очереди;
- `IN_SERVICE` — заявка обслуживается;
- `SERVED` — обслуживание завершено;
- `ABANDONED` — клиент ушёл, не дождавшись обслуживания.

## 7.3. Пакет `engine`

### `SimEvent`

Это минимальная сущность для очереди событий.

Поля:

- `time` — момент наступления события;
- `type` — тип события;
- `request` — ссылка на заявку для события прибытия;
- `operatorId` — оператор для события завершения обслуживания.

Поддерживаемые типы:

- `ARRIVAL`;
- `DEPARTURE`.

Сортировка событий выполняется по времени через `Comparable<SimEvent>`.

### `CallCenterEngine`

Это центральный класс проекта. Он полностью управляет жизненным циклом симуляции.

#### Внутреннее состояние движка

| Поле | Назначение |
| --- | --- |
| `config` | активная конфигурация симуляции |
| `random` | генератор случайных чисел |
| `currentTime` | текущее время модели |
| `currentIntervalIndex` | индекс текущего интервала |
| `lastRequestId` | счётчик идентификаторов заявок |
| `lastOperatorId` | счётчик идентификаторов операторов |
| `eventQueue` | приоритетная очередь будущих событий |
| `requestQueue` | очередь ожидающих заявок |
| `activeOperators` | операторы, которые ещё могут принимать вызовы |
| `allOperators` | все созданные за прогон операторы |
| `allRequests` | все созданные за прогон заявки |

#### Публичные методы

| Метод | Роль |
| --- | --- |
| `CallCenterEngine(SimulationConfig config)` | создаёт движок и фиксирует `seed` |
| `initialize()` | подготавливает первую волну операторов и заявок |
| `runNextInterval()` | отрабатывает один временной интервал |
| `getSnapshot()` | возвращает краткое состояние модели после шага |
| `generateResults()` | строит итоговые агрегированные метрики |

#### Внутренние методы и их логика

| Метод | Что делает |
| --- | --- |
| `generateArrivalsForInterval(int intervalIndex)` | генерирует все `ARRIVAL`-события внутри интервала |
| `processArrival(CallRequest req)` | пытается сразу выдать заявку оператору или ставит в очередь |
| `processDeparture(int operatorId)` | завершает обслуживание, освобождает оператора и при необходимости выдаёт ему новую заявку |
| `assignRequest(CallRequest req, CallOperator op)` | назначает заявку оператору и планирует событие `DEPARTURE` |
| `lazyQueueCleanup()` | удаляет из очереди клиентов, чей лимит ожидания истёк |
| `findOptimalOperator()` | ищет свободного оператора с минимальным `releaseTime` |
| `addOperatorsForInterval(int intervalIndex)` | добавляет новых операторов по расписанию |
| `distributeOperatorBusyTime(CallOperator op, double start, double end)` | раскладывает занятость оператора по интервалам |
| `finalizeRemainingEvents()` | дорабатывает хвост событий после последнего интервала |

#### Алгоритм `initialize()`

1. Сбрасывает время модели в `0.0`.
2. Сбрасывает индекс интервала в `0`.
3. Добавляет операторов для первого интервала.
4. Генерирует поступления заявок для первого интервала.

#### Алгоритм `runNextInterval()`

1. Проверяет, не закончены ли все интервалы.
2. Вычисляет конец текущего интервала.
3. Последовательно извлекает из `eventQueue` все события, время которых не превышает конец интервала.
4. Для `ARRIVAL` вызывает `processArrival`.
5. Для `DEPARTURE` вызывает `processDeparture`.
6. Перемещает `currentTime` к концу интервала.
7. Увеличивает индекс интервала.
8. Если интервалы ещё есть, добавляет операторов и новые заявки для следующего интервала.
9. Если интервал был последним, вызывает `finalizeRemainingEvents()`.

#### Поведение `assignRequest()`

При назначении заявки оператору движок:

1. Вычисляет длительность обслуживания.
2. Проверяет, может ли оператор завершить обслуживание до конца смены.
3. Если не может:
   оператор удаляется из `activeOperators`, заявка возвращается в начало очереди и вызывается `processDeparture()` для этого оператора.
4. Если может:
   заявка переводится в `IN_SERVICE`, оператор получает текущую заявку, занятость распределяется по интервалам и в очередь событий добавляется `DEPARTURE`.

Из этого следует важная особенность: оператор исключается из активного списка, как только становится ясно, что новую заявку до конца смены он уже не закончит.

#### Поведение `lazyQueueCleanup()`

Метод проходит по очереди ожидающих заявок и сравнивает:

```text
currentTime >= appearanceTime + patienceTime
```

Если условие истинно, заявка получает статус `ABANDONED` и удаляется из очереди.

#### Итог `generateResults()`

Метод:

- разбивает заявки на обслуженные и потерянные;
- считает суммарные показатели;
- строит средние значения;
- рассчитывает поинтервальные коэффициенты `K_client`, `K_oper` и целевую функцию.

## 7.5. Пакет `result`

### `SimulationSnapshot`

Класс хранит краткое состояние модели после очередного шага:

- `currentTime` — текущее время;
- `currentInterval` — номер текущего интервала;
- `queueLength` — длина очереди;
- `activeOperatorsCount` — число активных операторов;
- `busyOperatorsCount` — число занятых операторов;
- `totalProcessed` — количество обслуженных заявок;
- `totalAbandoned` — количество потерянных заявок.

### `SimulationResults`

Класс является контейнером итоговых данных симуляции.

Поля:

- `allRequests` — список всех заявок;
- `allOperators` — список всех операторов;
- `kClientByInterval` — `K_client` по интервалам;
- `kOperByInterval` — `K_oper` по интервалам;
- `targetFunctionByInterval` — значение целевой функции по интервалам;
- `totalArrived` — всего заявок поступило;
- `totalServed` — успешно обслужено;
- `totalAbandoned` — ушли из-за ожидания;
- `totalUnresolved` — остались в системе к завершению модели;
- `averageQueueTime` — среднее ожидание;
- `averageServiceTime` — среднее время обслуживания;
- `serviceProbability` — вероятность обслуживания;
- `abandonmentProbability` — вероятность ухода.

## 7.7. Пакет `demo`

### `Main`

Это реальная точка входа в проект.

Что делает `demo.Main`:

1. Создаёт `SimulationConfig`.
2. Инициализирует `Random` на `config.seed`.
3. Формирует список `lambdaByInterval`.
4. Записывает расписание операторов.
5. Запускает полный прогон через `AnyLogicAdapter`.
6. Определяет путь к отчёту:
   если путь передан аргументом, используется он; иначе берётся путь по умолчанию.
7. Сохраняет отчёт через `SimulationReportWriter`.
8. Печатает краткую итоговую статистику в консоль.

### Генерация `lambdaByInterval` в демонстрации

Демонстрационный сценарий не хранит список интенсивностей в константе, а строит его на лету:

- базовое значение равно `100`;
- первое значение добавляется как `100 / 3600.0`;
- далее 35 раз новое значение получается формулой `rand.nextInt(33) + (lambdaBase - 16)`;
- итоговое значение для конфигурации переводится в "заявок в секунду" делением на `3600.0`.

Это означает, что демонстрационный поток изменяется случайно, но воспроизводимо при фиксированном `seed`.

## 7.8. Файл `App.java`

`src/App.java` содержит шаблонный вывод `Hello, World!` и не связан с симуляцией. Документировать его как часть модели не требуется, но важно понимать, что это не альтернативная точка входа для расчёта колл-центра.

## 8. Подробный жизненный цикл заявки

Ниже описан путь одной заявки через систему.

1. Во время генерации поступлений создаётся `CallRequest`.
2. В очередь событий добавляется `SimEvent` типа `ARRIVAL`.
3. Когда событие наступает, движок пытается найти свободного оператора.
4. Если оператор найден и его смена позволяет завершить обслуживание, заявка сразу переходит в `IN_SERVICE`.
5. Если свободного подходящего оператора нет, заявка ставится в очередь ожидания.
6. При каждом удобном моменте вызывается `lazyQueueCleanup()`, и часть ожидающих заявок может стать `ABANDONED`.
7. Если оператор освобождается до того, как клиент ушёл, заявка начинает обслуживание.
8. После окончания обслуживания заявка получает статус `SERVED`.

## 9. Подробный жизненный цикл оператора

1. В начале заданного интервала движок создаёт нового `CallOperator`.
2. Оператор сразу попадает в список `activeOperators`.
3. Пока оператор свободен и смена не закончилась, он может принимать новые заявки.
4. После назначения заявки оператор помечается занятым до `releaseTime`.
5. При событии `DEPARTURE` оператор освобождается.
6. Если его смена уже закончилась, оператор удаляется из `activeOperators`.
7. Если смена ещё не закончилась, он либо берёт следующую заявку, либо остаётся свободным.

## 10. Расчёт итоговых метрик

### 10.1. Базовые счётчики

```text
totalArrived = количество всех созданных заявок
totalServed = количество заявок со статусом SERVED
totalAbandoned = количество заявок со статусом ABANDONED
totalUnresolved = totalArrived - totalServed - totalAbandoned
```

### 10.2. Средние значения

Среднее ожидание считается по заявкам, которые уже завершили путь, то есть:

- обслужены;
- или покинули очередь.

```text
averageQueueTime = average(queueTime по resolvedRequests)
```

Среднее время обслуживания считается только по обслуженным заявкам:

```text
averageServiceTime = average(leavingTime - serviceStartTime по servedRequests)
```

### 10.3. Вероятности

```text
serviceProbability = totalServed / totalArrived
abandonmentProbability = totalAbandoned / totalArrived
```

Если заявок не было, обе вероятности принимаются равными `0.0`.

### 10.4. Коэффициент клиента `K_client`

Для каждого интервала берутся только завершившие путь заявки, поступившие в этом интервале:

```text
avgQueue_i = average(queueTime по intervalReqs)
K_client_i = avgQueue_i * nu
```

Если завершившихся заявок в интервале нет, коэффициент равен `0`.

### 10.5. Коэффициент оператора `K_oper`

Сначала считается суммарное занятое время всех операторов в интервале:

```text
totalBusyTime_i = sum(busyTimeByInterval[i] по всем операторам)
```

Затем определяется количество операторов, чья смена покрывает начало интервала:

```text
opsInInterval_i = count(shiftStartTime <= intervalStart && shiftEndTime > intervalStart)
```

После этого:

```text
K_oper_i = totalBusyTime_i / (opsInInterval_i * intervalLength)
```

Если активных операторов в интервале нет, коэффициент равен `0`.

### 10.6. Целевая функция

В коде жёстко заданы коэффициенты:

```text
C1 = 1.0
C2 = 1.0
```

Итог по интервалу:

```text
target_i = C1 * K_oper_i - C2 * K_client_i
```

Сейчас целевая функция одинаково взвешивает:

- выгодность загрузки операторов;
- стоимость ожидания клиента.

## 11. Формат текстового отчёта

Метод `SimulationReportWriter.writeReport(...)` записывает отчёт в следующем порядке:

1. список `lambdaByInterval`;
2. список `operatorsSchedule`;
3. подробности по каждой завершённой заявке;
4. значения `K_client` по интервалам;
5. занятость каждого оператора по интервалам;
6. коэффициент загрузки `K_oper` по интервалам;
7. целевая функция по интервалам;
8. итоговая сводка по прогону.

### 11.1. Что попадает в блок завершённых заявок

Для `SERVED` фиксируются:

- идентификатор заявки;
- идентификатор оператора;
- время в очереди;
- время появления;
- время ухода.

Для `ABANDONED` фиксируются:

- идентификатор заявки;
- время ожидания до ухода;
- время появления;
- время ухода.

### 11.2. Формат чисел

Числа форматируются через шаблон:

```text
0.#######
```

Разделитель дробной части принудительно задаётся как точка через `Locale.US`.

## 12. Особенности реализации, которые важно знать

### 12.1. Операторы добавляются, а не задаются общим количеством

`operatorsSchedule[i]` означает количество новых операторов, которые начинают смену в интервале `i`. Это не число операторов, одновременно работающих в интервале.

### 12.2. Нет отдельного события отказа клиента

Клиенты уходят не по самостоятельному событию в очереди событий. Отказ фиксируется только при очередной проверке очереди. Из-за этого логика проще, но поведение важно учитывать при расширении модели.

### 12.3. Идентификатор заявки используется как индекс

В `processDeparture()` завершённая заявка извлекается так:

```text
allRequests.get(op.getCurrentRequestId() - 1)
```

Это безопасно только потому, что:

- идентификаторы идут подряд от `1`;
- заявки добавляются в `allRequests` строго в порядке создания.

### 12.4. Возможны незавершённые заявки

`totalUnresolved` может быть больше нуля, если к концу симуляции часть заявок:

- всё ещё ждёт в очереди;
- ещё не превысила своё время терпения;
- либо не успела дойти до состояния обслуживания или отказа.

### 12.5. Для `lambda = 0` код не защищён

В `generateArrivalsForInterval()` используется деление на `lambda`. Если передать `0`, логика генерации межприходовых интервалов станет некорректной. Конфигурация должна обеспечивать положительные значения интенсивности.

## 13. Структура каталогов и среда разработки

Согласно `.vscode/settings.json`:

- исходники находятся в `src`;
- выходная директория по умолчанию — `bin`;
- внешние библиотеки ищутся в `lib/**/*.jar`.

На момент документирования:

- папка `lib` пуста;
- проект можно считать зависимым только от JDK;
- артефакты `bin` являются производными и не должны использоваться как источник документации.

## 14. Как запускать и расширять проект

### 14.1. Минимальный сценарий запуска

1. Создать `SimulationConfig`.
2. Заполнить `lambdaByInterval`.
3. Заполнить `operatorsSchedule`.
4. Передать конфигурацию в `AnyLogicAdapter.runFastFullSimulation(...)`.
5. Получить `SimulationResults`.
6. При необходимости записать отчёт через `SimulationReportWriter`.

### 14.2. Если нужно запускать модель пошагово

Можно использовать фасад `AnyLogicAdapter`:

1. `initSimulation(config)`;
2. повторять `stepInterval()`;
3. после каждого шага читать `getCurrentSnapshot()`;
4. в конце вызвать `getFinalResults()`.

### 14.3. Если нужно развивать модель дальше

Наиболее естественные точки расширения:

- добавить отдельный тип события для отказа клиента;
- вынести веса `C1` и `C2` в `SimulationConfig`;
- добавить несколько очередей или приоритеты клиентов;
- разделить флаг случайности терпения и обслуживания на два независимых параметра;
- ввести валидацию конфигурации перед запуском;
- сделать настоящий слой интеграции с AnyLogic или UI.

## 15. Краткий итог

`CallCenterJava` — это компактная, но полноценная событийная модель колл-центра. Основная ценность проекта состоит в том, что он уже разделён на:

- конфигурацию;
- доменную модель;
- движок симуляции;
- фасад интеграции;
- результат;
- экспорт отчёта.
