# Документация проекта CallCenterJava

## 1. Назначение проекта

`CallCenterJava` моделирует работу колл-центра методом дискретно-событийной симуляции. Проект нужен для того, чтобы по входному профилю нагрузки оценить:

- сколько заявок будет обслужено;
- сколько клиентов уйдёт из очереди;
- насколько загружены операторы;
- устойчива ли система на отдельных интервалах;
- как выбранное количество операторов влияет на качество обслуживания.

Текущая версия проекта ориентирована на сценарий "загрузить датасет, автоматически подготовить конфигурацию, выполнить полный прогон и сохранить текстовый отчёт".

## 2. Предметная область и правила модели

### 2.1. Что считается заявкой

Заявка (`CallRequest`) представляет одно входящее обращение клиента. Для неё фиксируются:

- момент появления в системе;
- интервал, в котором она появилась;
- максимально допустимое время ожидания;
- фактическое время ожидания;
- оператор, начавший обслуживание;
- момент ухода из системы;
- итоговый статус.

Статусы заявки описываются перечислением `RequestStatus`:

- `WAITING`;
- `IN_SERVICE`;
- `SERVED`;
- `ABANDONED`.

### 2.2. Что считается оператором

Оператор (`CallOperator`) начинает смену в некоторый момент времени и работает фиксированную длительность `shiftLength`. Оператор:

- может быть свободным или занятым;
- обслуживает только одну заявку одновременно;
- может принять новую заявку только если успевает завершить разговор до конца смены;
- после освобождения либо получает следующую заявку из очереди, либо удаляется из активного списка, если смена уже закончилась.

### 2.3. Как трактуется очередь

Очередь реализована как `LinkedList<CallRequest>` и обслуживается по строгому FIFO-принципу:

- необслуженные заявки добавляются в хвост;
- назначение всегда начинается с головы очереди;
- если головную заявку нельзя назначить в текущий момент, движок не перескакивает к более поздним заявкам.

### 2.4. Что считается уходом клиента

Если клиент ждал дольше своего допустимого времени ожидания `patienceTime`, заявка получает статус `ABANDONED`.

В модели нет отдельного события "клиент ушёл". Отказ обрабатывается лениво через `lazyQueueCleanup()`, когда движок:

- обрабатывает новое поступление;
- пытается разобрать очередь;
- завершает симуляцию.

### 2.5. Как формируется входящий поток

Движок поддерживает два режима генерации заявок.

1. Через `lambdaByInterval`.
   Для интервала генерируется пуассоновский поток с экспоненциальными межприходами.

2. Через `arrivalsByInterval`.
   Для интервала задаётся точное число заявок, а их моменты появления случайно распределяются внутри интервала и затем сортируются.

Если заданы `arrivalsByInterval`, именно они используются для фактической генерации заявок.

## 4. Жизненный цикл приложения

Актуальный сценарий работы выглядит так:

```text
demo.Main
  -> определяет путь к dataset.json и report.txt
  -> DatasetConfigLoader.load(datasetPath)
       -> читает JSON
       -> строит SimulationConfig
       -> заполняет totalOperators, totalIntervals, intervalLength, arrivalsByInterval
  -> AnyLogicAdapter.runFastFullSimulation(config)
       -> new CallCenterEngine(config)
       -> engine.initialize()
            -> validateCoreConfig()
            -> ensureArrivalsByInterval()
            -> ensureLambdaByInterval()
            -> ensureOperatorsSchedule()
       -> пока engine.runNextInterval() == true
       -> engine.generateResults()
  -> SimulationReportWriter.writeReport(config, results, reportPath)
  -> печать итоговой сводки в консоль
```

Практически это значит, что часть конфигурации подготавливается не в `Main`, а уже внутри движка.

## 5. Конфигурация симуляции и входные данные

Класс `config.SimulationConfig` хранит все параметры одного прогона.

| Поле | Значение по умолчанию | Единицы | Смысл |
| --- | --- | --- | --- |
| `mu` | `1.0 / 240.0` | 1/сек | интенсивность обслуживания одним оператором |
| `nu` | `1.0 / 240.0` | 1/сек | интенсивность потери терпения клиента |
| `intervalLength` | `1200.0` | сек | длительность одного интервала |
| `totalIntervals` | `36` | шт | число интервалов в модели |
| `shiftLength` | `14400.0` | сек | длительность смены одного оператора |
| `lambdaByInterval` | `null` | 1/сек | интенсивность входящего потока по интервалам |
| `arrivalsByInterval` | `null` | шт | точное число заявок по интервалам |
| `operatorsSchedule` | `null` | шт | число новых операторов, стартующих в интервале |
| `totalOperators` | `0` | шт | общее доступное число операторов для расписания |
| `seed` | `System.currentTimeMillis()` | число | зерно генератора случайных чисел |
| `useRandomServiceTime` | `true` | флаг | включает случайные времена обслуживания и терпения |

### 5.1. Что заполняется пользователем, а что может достроиться автоматически

В типичном сценарии из `dataset.json` приходят:

- `totalOperators`;
- `totalIntervals`;
- `intervalLength`;
- `arrivalsByInterval`.

Во время `initialize()` движок при необходимости достраивает:

- `lambdaByInterval` на основе `arrivalsByInterval`;
- `operatorsSchedule` на основе `lambdaByInterval`, `mu`, `shiftLength` и `totalOperators`.

### 5.2. Валидация конфигурации

Перед запуском движок проверяет:

- `totalOperators >= 0`;
- `totalIntervals >= 0`;
- `intervalLength > 0`;
- `shiftLength > 0`;
- `mu > 0`;
- `nu > 0`;
- корректность размеров `arrivalsByInterval`, `lambdaByInterval` и `operatorsSchedule`;
- отсутствие отрицательных значений в этих списках;
- что сумма `operatorsSchedule` не превышает `totalOperators`, если расписание задано явно.

### 5.3. Формат входного JSON-датасета

`DatasetConfigLoader` ожидает JSON-объект вида:

```json
{
  "totalOperators": 10,
  "schedule": [
    { "timeFromStartMs": 0, "callCount": 2 },
    { "timeFromStartMs": 1200000, "callCount": 4 }
  ]
}
```

Обязательные поля:

- `totalOperators` — общее количество доступных операторов;
- `schedule` — массив интервалов.

Для каждого элемента `schedule` используются:

- `timeFromStartMs` — старт интервала в миллисекундах от начала модели;
- `callCount` — число заявок в интервале.

Поле `timeString`, присутствующее в примере `dataset.json`, служит только для человека и загрузчиком не используется.

### 5.4. Что проверяет `DatasetConfigLoader`

Загрузчик дополнительно валидирует, что:

- корень документа — JSON-объект;
- `schedule` не пуст;
- первая запись начинается с `0 ms`;
- интервалы идут равномерным шагом;
- числовые поля действительно целочисленные;
- все элементы `schedule` являются объектами.

Если в датасете только один интервал, его длина берётся из значения по умолчанию `SimulationConfig.intervalLength`.

## 6. Математическая и алгоритмическая модель

### 6.1. Поступление заявок через `lambdaByInterval`

Если используется интенсивность `lambda`, межприходовое время вычисляется как:

```text
delta = -ln(1 - u) / lambda
```

где `u` — равномерная случайная величина на отрезке `(0, 1)`.

### 6.2. Поступление заявок через `arrivalsByInterval`

Если используется точное число заявок, движок:

1. берёт `arrivalsCount` для интервала;
2. генерирует `arrivalsCount` случайных моментов равномерно внутри интервала;
3. сортирует их;
4. планирует события `ARRIVAL`.

Такой режим сохраняет точное количество заявок по интервалам, но оставляет случайность внутри интервала.

### 6.3. Время терпения клиента

Если `useRandomServiceTime == true`, терпение клиента генерируется экспоненциально:

```text
patience = -ln(1 - u) / nu
```

Иначе:

```text
patience = 1 / nu
```

### 6.4. Время обслуживания

Если `useRandomServiceTime == true`, длительность обслуживания генерируется как:

```text
serviceTime = -ln(1 - u) / mu
```

Иначе:

```text
serviceTime = 1 / mu
```

Важно: в текущей реализации `serviceTime` сэмплируется в момент попытки назначения заявки оператору, а не фиксируется заранее в момент появления заявки.

### 6.5. Автоматическое построение расписания операторов

Если `operatorsSchedule == null`, движок строит его сам.

Общая идея:

1. оценивается потребность в операторах как `lambda / mu`;
2. определяется, сколько интервалов покрывает одна смена;
3. строится минимальное покрытие;
4. если операторов не хватает, включается best-effort эвристика, которая пытается максимизировать покрытый спрос при ограничении `totalOperators`.

Это значит, что `totalOperators` сейчас работает как верхняя граница ресурсов, а не как "число операторов в каждом интервале".

## 7. Подробное описание пакетов и классов

## 7.1. Пакет `config`

### `SimulationConfig`

`SimulationConfig` — это простой контейнер данных без собственной логики. Он хранит:

- параметры интенсивностей `mu` и `nu`;
- структуру временной сетки;
- входной поток через `lambdaByInterval` или `arrivalsByInterval`;
- расписание операторов или ограничения для его автопостроения;
- параметры воспроизводимости через `seed`.

Часть полей может быть изначально пустой и заполняться уже самим движком во время `initialize()`.

## 7.2. Пакет `input`

### `SimpleJsonParser`

Это встроенный минималистичный JSON-парсер, написанный без внешних зависимостей. Он умеет разбирать:

- объекты;
- массивы;
- строки;
- числа;
- `true`, `false`, `null`;
- стандартные escape-последовательности, включая `\uXXXX`.

Он нужен только как низкоуровневая утилита для загрузки датасета.

### `DatasetConfigLoader`

Класс загружает внешний JSON-датасет и преобразует его в `SimulationConfig`.

Что делает `DatasetConfigLoader.load(...)`:

1. читает файл в UTF-8;
2. разбирает JSON;
3. извлекает `totalOperators` и массив `schedule`;
4. сортирует интервалы по `timeFromStartMs`;
5. определяет `intervalLength`;
6. строит `arrivalsByInterval`;
7. возвращает конфигурацию, готовую к дальнейшей нормализации в движке.

### `dataset.json`

Файл `src/input/dataset.json` — это пример входного сценария с 36 интервалами и общим ограничением `totalOperators = 10`.

## 7.3. Пакет `model`

### `CallRequest`

Класс описывает одну заявку.

Состояние объекта:

- `id` — уникальный идентификатор заявки;
- `appearanceTime` — время появления заявки;
- `patienceTime` — максимум ожидания перед уходом;
- `intervalIndex` — интервал поступления;
- `queueTime` — фактическое время ожидания;
- `status` — текущий статус из `RequestStatus`;
- `operatorId` — оператор, начавший обслуживание;
- `serviceStartTime` — момент начала обслуживания;
- `leavingTime` — момент выхода из системы.

Основные методы:

| Метод | Назначение |
| --- | --- |
| `startService(double currentTime, int operatorId)` | переводит заявку в `IN_SERVICE`, фиксирует время ожидания и оператора |
| `completeService(double currentTime)` | переводит заявку в `SERVED` и записывает время выхода |
| `abandon(double currentTime)` | переводит заявку в `ABANDONED`, фиксирует уход и время ожидания |

Правила переходов:

```text
WAITING -> IN_SERVICE -> SERVED
WAITING -> ABANDONED
```

### `CallOperator`

Класс описывает одного оператора.

Состояние объекта:

- `id` — идентификатор оператора;
- `shiftStartTime` — начало смены;
- `shiftEndTime` — конец смены;
- `busy` — признак занятости;
- `currentRequestId` — текущая заявка;
- `releaseTime` — момент освобождения;
- `busyTimeByInterval` — карта занятости по интервалам.

Основные методы:

| Метод | Назначение |
| --- | --- |
| `canTakeRequest(double currentTime, double estimatedServiceTime)` | проверяет, успеет ли оператор закончить заявку до конца смены |
| `assignRequest(int requestId, double currentTime, double serviceTime)` | помечает оператора занятым и записывает время освобождения |
| `freeUp()` | освобождает оператора после завершения обслуживания |
| `addBusyTime(int interval, double time)` | добавляет накопленное занятое время в указанный интервал |

### `RequestStatus`

Перечисление описывает четыре состояния заявки:

- `WAITING` — заявка находится в очереди;
- `IN_SERVICE` — заявка обслуживается;
- `SERVED` — обслуживание завершено;
- `ABANDONED` — клиент ушёл, не дождавшись обслуживания.

## 7.4. Пакет `engine`

### `SimEvent`

Это минимальная сущность для приоритетной очереди событий.

Поля:

- `time` — момент наступления события;
- `type` — тип события;
- `request` — ссылка на заявку для события прибытия;
- `operatorId` — оператор для события завершения обслуживания.

Поддерживаемые типы:

- `ARRIVAL`;
- `DEPARTURE`.

Сортировка событий выполняется по времени через `Comparable<SimEvent>`.

### `CallCenterEngine`

Это центральный класс проекта. Он:

- валидирует и нормализует конфигурацию;
- генерирует входящий поток;
- достраивает расписание операторов при необходимости;
- обрабатывает события `ARRIVAL` и `DEPARTURE`;
- управляет очередью ожидания;
- собирает агрегированные результаты.

#### Внутреннее состояние движка

| Поле | Назначение |
| --- | --- |
| `config` | активная конфигурация симуляции |
| `random` | генератор случайных чисел |
| `currentTime` | текущее время модели |
| `currentIntervalIndex` | индекс текущего интервала |
| `lastRequestId` | счётчик идентификаторов заявок |
| `lastOperatorId` | счётчик идентификаторов операторов |
| `eventQueue` | приоритетная очередь будущих событий |
| `requestQueue` | очередь ожидающих заявок |
| `activeOperators` | операторы, которые ещё могут принимать вызовы |
| `allOperators` | все созданные за прогон операторы |
| `allRequests` | все созданные за прогон заявки |

#### Публичные методы

| Метод | Роль |
| --- | --- |
| `CallCenterEngine(SimulationConfig config)` | создаёт движок и инициализирует генератор случайности |
| `initialize()` | валидирует и нормализует конфиг, затем подготавливает первый интервал |
| `runNextInterval()` | отрабатывает один временной интервал |
| `getSnapshot()` | возвращает краткое состояние модели после шага |
| `generateResults()` | строит итоговые агрегированные метрики |

#### Основные внутренние методы

| Метод | Что делает |
| --- | --- |
| `prepareSimulationConfig()` | запускает полную подготовку конфигурации |
| `ensureLambdaByInterval()` | валидирует `lambdaByInterval` или строит его из `arrivalsByInterval` |
| `ensureOperatorsSchedule()` | валидирует расписание операторов или строит его автоматически |
| `buildOperatorsSchedule()` | рассчитывает авто-расписание по нагрузке и лимиту операторов |
| `generateArrivalsForInterval(int intervalIndex)` | выбирает режим генерации заявок для интервала |
| `generateFixedArrivalsForInterval(...)` | создаёт точное число заявок внутри интервала |
| `processArrival(CallRequest req)` | пытается сразу назначить заявку или ставит её в очередь |
| `processDeparture(int operatorId)` | завершает разговор, освобождает оператора и запускает разбор очереди |
| `tryAssignRequest(CallRequest req)` | подбирает оператора и длительность обслуживания |
| `assignRequestToOperator(...)` | назначает заявку оператору и планирует `DEPARTURE` |
| `assignQueuedRequests()` | разбирает очередь ожидания по FIFO |
| `lazyQueueCleanup()` | удаляет из очереди клиентов, чей лимит ожидания истёк |
| `findOptimalOperator(double serviceTime)` | ищет подходящего свободного оператора |
| `removeExpiredIdleOperators()` | удаляет простаивающих операторов после конца смены |
| `addOperatorsForInterval(int intervalIndex)` | добавляет новых операторов по расписанию |
| `distributeOperatorBusyTime(CallOperator op, double start, double end)` | раскладывает занятость оператора по интервалам |
| `finalizeRemainingEvents()` | дорабатывает хвост событий после последнего интервала |

Подробная логика `CallCenterEngine` вынесена в отдельный документ: `docs/CALL_CENTER_ENGINE_DOCUMENTATION.md`.

## 7.5. Пакет `integration`

Название класса говорит об интеграции с AnyLogic, но в текущей версии это просто удобный фасад над `CallCenterEngine`.

Методы:

| Метод | Назначение |

Класс полезен как точка расширения, если в будущем понадобится:

- подключение GUI;
- внешний контроллер пошагового исполнения;
- интеграция с другим средством моделирования.

## 7.6. Пакет `result`

### `SimulationSnapshot`

Класс хранит краткое состояние модели после очередного шага:

- `currentTime` — текущее время;
- `currentInterval` — номер текущего интервала;
- `queueLength` — длина очереди;
- `activeOperatorsCount` — число активных операторов;
- `busyOperatorsCount` — число занятых операторов;
- `totalProcessed` — количество обслуженных заявок;
- `totalAbandoned` — количество потерянных заявок.

### `SimulationResults`

Класс является контейнером итоговых данных симуляции.

Поля:

- `allRequests` — список всех заявок;
- `allOperators` — список всех операторов;
- `kClientByInterval` — `K_client` по интервалам;
- `kOperByInterval` — `K_oper` по интервалам;
- `stabilityByInterval` — коэффициент устойчивости `K_уст` по интервалам;
- `operatorsCountByInterval` — число операторов, работающих в интервале;
- `serviceCapacityByInterval` — суммарная интенсивность обслуживания `N_оп * mu` по интервалам;
- `targetFunctionByInterval` — значение целевой функции по интервалам;
- `totalArrived` — всего заявок поступило;
- `totalServed` — успешно обслужено;
- `totalAbandoned` — ушли из-за ожидания;
- `totalUnresolved` — остались в системе к завершению модели;
- `averageQueueTime` — среднее ожидание;
- `averageServiceTime` — среднее время обслуживания;
- `serviceProbability` — вероятность обслуживания;
- `abandonmentProbability` — вероятность ухода;
- `minimumStability` — минимальный коэффициент устойчивости среди интервалов с ненулевым входящим потоком;
- `systemStable` — общий флаг устойчивости модели.

## 8. Подробный жизненный цикл заявки

Ниже описан путь одной заявки через систему.

1. Во время генерации поступлений создаётся `CallRequest`.
2. В очередь событий добавляется `SimEvent` типа `ARRIVAL`.
3. Когда событие наступает, `processArrival()` сначала запускает `lazyQueueCleanup()`.
4. Затем движок пытается сразу назначить заявку через `tryAssignRequest()`.
5. Если подходящего оператора нет, заявка попадает в хвост `requestQueue`.
6. При каждом удобном моменте очередь снова проверяется, и часть заявок может стать `ABANDONED`.
7. Когда оператор освобождается или начинается новая смена, движок пытается разобрать очередь через `assignQueuedRequests()`.
8. Если заявка получает оператора, она переходит в `IN_SERVICE`.
9. После события `DEPARTURE` заявка получает статус `SERVED`.

## 9. Подробный жизненный цикл оператора

1. В начале заданного интервала движок создаёт нового `CallOperator`.
2. Оператор сразу попадает в список `activeOperators`.
3. Пока оператор свободен и смена не закончилась, он может принимать новые заявки.
4. При попытке назначения проверяется, успеет ли он завершить разговор до `shiftEndTime`.
5. После назначения заявки оператор помечается занятым до `releaseTime`.
6. Во время разговора его занятость раскладывается по интервалам через `busyTimeByInterval`.
7. При событии `DEPARTURE` оператор освобождается.
8. Если смена уже закончилась и оператор простаивает, `removeExpiredIdleOperators()` убирает его из `activeOperators`.
9. Если смена ещё не закончилась, оператор либо получает следующую заявку из очереди, либо остаётся свободным.

## 10. Расчёт итоговых метрик

### 10.1. Базовые счётчики

```text
totalArrived = количество всех созданных заявок
totalServed = количество заявок со статусом SERVED
totalAbandoned = количество заявок со статусом ABANDONED
totalUnresolved = totalArrived - totalServed - totalAbandoned
```

### 10.2. Средние значения

```text
resolvedRequests = servedRequests + abandonedRequests
averageQueueTime = average(queueTime по resolvedRequests)
averageServiceTime = average(leavingTime - serviceStartTime по servedRequests)
```

### 10.3. Вероятности

```text
serviceProbability = totalServed / totalArrived
abandonmentProbability = totalAbandoned / totalArrived
```

Если заявок не было, обе вероятности принимаются равными `0.0`.

### 10.4. Коэффициент клиента `K_client`

Для каждого интервала берутся только завершившие путь заявки, поступившие в этом интервале:

```text
avgQueue_i = average(queueTime по intervalReqs)
K_client_i = avgQueue_i * nu
```

Если завершившихся заявок в интервале нет, коэффициент равен `0`.

### 10.5. Коэффициент оператора `K_oper`

Сначала считается суммарное занятое время всех операторов в интервале:

```text
totalBusyTime_i = sum(busyTimeByInterval[i] по всем операторам)
```

Затем определяется количество операторов, чья смена покрывает начало интервала:

```text
opsInInterval_i = count(shiftStartTime <= intervalStart && shiftEndTime > intervalStart)
```

После этого:

```text
K_oper_i = totalBusyTime_i / (opsInInterval_i * intervalLength)
```

Если активных операторов в интервале нет, коэффициент равен `0`.

### 10.6. Коэффициент устойчивости системы `K_уст`

Для каждого интервала движок дополнительно сохраняет две промежуточные величины:

```text
operatorsCountByInterval_i = opsInInterval_i
serviceCapacityByInterval_i = opsInInterval_i * mu
```

Сам коэффициент устойчивости вычисляется так:

```text
K_уст_i = (opsInInterval_i * mu) / lambda_i
```

Интерпретация результата:

- если `K_уст_i >= 1`, система в интервале считается устойчивой;
- если `K_уст_i < 1`, входящий поток превышает доступную интенсивность обслуживания;
- если `lambda_i <= 0`, метод `calculateStability(...)` возвращает `INF`.

По завершении расчётов дополнительно формируются:

- `minimumStability` — минимум по всем интервалам, где `lambda_i > 0`;
- `systemStable` — `true`, если для всех интервалов с `lambda_i > 0` выполнено `K_уст_i >= 1`.

### 10.7. Целевая функция

В коде жёстко заданы коэффициенты:

```text
C1 = 1.0
C2 = 1.0
```

Итог по интервалу:

```text
target_i = C1 * K_oper_i - C2 * K_client_i
```

Сейчас целевая функция одинаково взвешивает:

- выгодность загрузки операторов;
- стоимость ожидания клиента.

## 11. Формат текстового отчёта

Метод `SimulationReportWriter.writeReport(...)` записывает отчёт в следующем порядке:

1. список `lambdaByInterval`;
2. список `operatorsSchedule`;
3. число операторов, работавших по интервалам;
4. подробности по каждой завершённой заявке;
5. значения `K_client` по интервалам;
6. занятость каждого оператора по интервалам;
7. коэффициент загрузки `K_oper` по интервалам;
8. коэффициент устойчивости `K_уст` по интервалам;
9. отдельный блок с подробным расчётом устойчивости системы;
10. целевая функция по интервалам;
11. итоговая сводка по прогону.

### 11.1. Что попадает в блок завершённых заявок

Для `SERVED` выводятся:

- идентификатор заявки;
- идентификатор оператора;
- время в очереди;
- время появления;
- время ухода.

Для `ABANDONED` выводятся:

- идентификатор заявки;
- фактическое время ожидания до ухода;
- время появления;
- время ухода.

### 11.2. Что попадает в блок подробного расчёта устойчивости

В аналитическом блоке по каждому интервалу фиксируются:

- значение `lambda`;
- число активных операторов `N_оп`;
- суммарная интенсивность обслуживания `N_оп * mu`;
- подстановка в формулу `K_уст = (N_оп * mu) / lambda`;
- запас устойчивости `K_уст - 1`;
- текстовый вывод `устойчива` или `неустойчива`.

В начале блока дополнительно выводятся:

- критерий устойчивости `K_уст >= 1`;
- минимальный `K_уст` по модели;
- общий вывод по всей системе.

### 11.3. Формат чисел

Числа форматируются через шаблон:

```text
0.#######
```

Разделитель дробной части принудительно задаётся как точка через `Locale.US`.

## 12. Особенности реализации, которые важно знать

### 12.1. Конфигурация частично достраивается внутри движка

Если в `SimulationConfig` не заданы `lambdaByInterval` или `operatorsSchedule`, движок может построить их сам во время `initialize()`.

### 12.2. Нет отдельного события отказа клиента

Клиенты уходят не по самостоятельному событию в очереди событий. Отказ фиксируется только при очередной проверке очереди. Из-за этого логика проще, но поведение важно учитывать при расширении модели.

### 12.3. `arrivalsByInterval` имеет приоритет над `lambdaByInterval` для генерации заявок

Если заданы оба поля, реальные `ARRIVAL`-события строятся по `arrivalsByInterval`, а `lambdaByInterval` остаётся источником для аналитики и автопостроения расписания.

### 12.4. Идентификатор заявки используется как индекс

В `processDeparture()` завершённая заявка извлекается так:

```text
allRequests.get(op.getCurrentRequestId() - 1)
```

Это безопасно только потому, что:

- идентификаторы идут подряд от `1`;
- заявки добавляются в `allRequests` строго в порядке создания.

### 12.5. Возможны незавершённые заявки

`totalUnresolved` может быть больше нуля, если к концу симуляции часть заявок:

- всё ещё ждёт в очереди;
- ещё не превысила своё время терпения;
- либо не успела дойти до состояния обслуживания или отказа.

### 12.6. Для `lambda = 0` генерация заявок корректно пропускается, но устойчивость станет `INF`

Если `lambda <= 0`, новые заявки через режим `lambdaByInterval` не создаются, а `calculateStability(...)` возвращает `INF`.

### 12.7. Длительность обслуживания переоценивается на каждой попытке назначения

Пока заявка стоит в очереди, при каждой новой попытке назначения ей заново генерируется `serviceTime`. Это влияет на решение "успеет ли оператор до конца смены".

### 12.8. Один экземпляр `CallCenterEngine` не рассчитан на повторный независимый прогон

`initialize()` не очищает накопленные коллекции и счётчики, поэтому для нового прогона нужно создавать новый экземпляр движка.

## 14. Как запускать и расширять проект

### 14.1. Минимальный сценарий запуска

Сборка в PowerShell:

```powershell
javac -encoding UTF-8 -d bin (Get-ChildItem src -Recurse -Filter *.java | Select-Object -ExpandProperty FullName)
```

Запуск с датасетом и путями по умолчанию:

```powershell
java -cp bin demo.Main
```

Запуск с явным датасетом и отчётом:

```powershell

```

После запуска программа:

- печатает краткую сводку в консоль;
- сохраняет подробный отчёт в текстовый файл.

### 14.2. Если нужно запускать модель пошагово

Можно использовать фасад `AnyLogicAdapter`:

1. создать или загрузить `SimulationConfig`;
2. вызвать `initSimulation(config)`;
3. повторять `stepInterval()`;
4. после каждого шага читать `getCurrentSnapshot()`;
5. в конце вызвать `getFinalResults()`.

### 14.3. Если нужно развивать модель дальше

Наиболее естественные точки расширения:

- добавить отдельный тип события для отказа клиента;
- вынести веса `C1` и `C2` в `SimulationConfig`;
- добавить несколько очередей или приоритеты клиентов;
- разделить флаг случайности терпения и обслуживания на два независимых параметра;
- добавить более явную валидацию сценария `totalIntervals > 0`;
- синхронизировать или запретить противоречивую передачу `arrivalsByInterval` и `lambdaByInterval`;
- закреплять `serviceTime` за заявкой в момент появления, если нужна более физичная модель;
- сделать настоящий слой интеграции с AnyLogic или UI.
